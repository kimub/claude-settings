## 역할 구분

- **이 스킬**: 매핑 규칙, 포맷 변환 지식, 정책 정의 (지식/규칙)
- **jira-publisher 에이전트**: 이 스킬의 규칙을 읽고 실제 Jira API 호출 수행 (실행)

## AI 작성 안내 텍스트 (필수)

모든 Jira 이슈의 Description 최상단에 아래 안내 문구를 반드시 삽입한다.
이 규칙은 신규 생성, 업데이트, 단일/복합 업로드 모두에 적용된다.

**Wiki 마크업 형식:**
```
{info}이 문서는 AI(Claude)가 작성하였으며, 내용의 정확성은 담당자의 검토가 필요합니다.{info}
```

**일반 텍스트 형식 (ADF 사용 시):**
```
[AI 작성] 이 문서는 AI(Claude)가 작성하였으며, 내용의 정확성은 담당자의 검토가 필요합니다.
```

> 안내 텍스트에 이모지를 사용하지 않는다. Jira UI에서 이모지 렌더링이 환경마다 다를 수 있고,
> 공식 문서의 톤에 맞지 않기 때문이다. `[AI 작성]` 같은 텍스트 태그를 사용한다.

## 필수 레이블 정책

모든 Jira 이슈 생성 시 `AI-Claude` 레이블을 반드시 포함한다.
AI가 작성한 이슈임을 레이블로 식별할 수 있게 하기 위함이다.
문서 유형별 레이블(pdca-plan, pdca-design 등)과 함께 항상 추가한다.

- 신규 생성: labels 필드에 `AI-Claude` + 문서 유형별 레이블
- 기존 이슈 업데이트: 기존 레이블에 `AI-Claude`가 없으면 추가

## 프로젝트 키 정책

- 프로젝트 키는 반드시 사용자에게 명시적으로 받아야 한다
- 자동 탐지, 추정, 캐싱된 값 재사용 금지
- 매 게시 요청마다 사용자에게 프로젝트 키를 질문한다
- 예시 질문: "어떤 Jira 프로젝트에 올릴까요? (프로젝트 키를 알려주세요, 예: MZ00006)"

## 문서 유형별 Jira 매핑 규칙

각 산출물은 Jira 이슈의 어느 필드에, 어떤 형식으로 올라가는지가 다르다.

### plan.md (계획서)
- 이슈 유형: Task 또는 Story
- Summary: "[Plan] {작업 제목}"
- Description에 포함할 섹션: 배경, 목적, 범위, 성공 기준
- Label: AI-Claude, pdca-plan

### design.md (설계서)
- 이슈 유형: Task 또는 Sub-task
- Summary: "[Design] {작업 제목}"
- Description에 포함할 섹션: 아키텍처, 구현 방안, 대상 시스템, 설정값
- Label: AI-Claude, pdca-design

### 작업계획서 (workplan/*.md)
- 이슈 유형: Task
- Summary: "[Workplan] {작업 제목}"
- Description에 포함할 섹션: 개요 테이블 + 작업 순서 요약 (전체 본문 아님)
- 첨부파일: PDF가 있으면 PDF 첨부, 없으면 마크다운 원본 첨부
- Label: AI-Claude, workplan

### 분석서/보고서
- 이슈 유형: Task
- Summary: "[Report] {작업 제목}" 또는 "[Analysis] {작업 제목}"
- Description에 포함할 섹션: 분석 결과 요약, 주요 발견사항
- Label: AI-Claude, pdca-report

## 복합 업로드 규칙

### plan.md + design.md 묶어서 올릴 때
- 단일 이슈로 생성
- Summary: "[PDCA] {작업 제목}"
- Description 구조:
  - {AI 작성 안내 텍스트}
  - h2. 1. 계획 (Plan)
  - {plan.md 핵심 내용}
  - h2. 2. 설계 (Design)
  - {design.md 핵심 내용}
- Label: AI-Claude, pdca-plan, pdca-design

### plan.md + design.md + 작업계획서 묶어서 올릴 때
- 부모 이슈 1개 + Sub-task 구조 권장
- 부모: "[PDCA] {작업 제목}" — plan + design 통합 내용
- Sub-task: "[Workplan] {작업 제목}" — 작업계획서 내용 + PDF 첨부
- 또는 사용자가 단일 이슈를 원하면 하나로 통합
- Label: AI-Claude, pdca-plan, pdca-design, workplan

## Description 작성 포맷

Atlassian MCP의 `createJiraIssue`/`editJiraIssue` API는 **Markdown** 형식의 description을 받는다.
MCP 내부에서 Markdown → ADF(Atlassian Document Format)로 자동 변환하여 Jira에 저장한다.

따라서 description은 **Markdown으로 작성**하면 된다 (Wiki 마크업 아님).

### Markdown 기본 문법 (그대로 사용 가능)

- `# 제목`, `## 소제목`, `### 소소제목`
- `**볼드**`, `*이탤릭*`
- `> 인용문`
- `- 리스트`, `1. 순서 리스트`
- 코드블록: 백틱 3개 감싸기
- 표: `| 헤더 | 헤더 |` + `| --- | --- |` + `| 셀 | 셀 |`

### Markdown→ADF 변환 함정 (반드시 회피)

Atlassian MCP의 Markdown→ADF 변환기는 테이블 셀 내 특정 패턴을 의도치 않게
인라인 코드(`<code>`)로 렌더링하는 특성이 있다.
API 응답의 markdown에서는 정상으로 보이지만, Jira UI에서 ADF 렌더링 시 코드 블록으로 표시된다.

**코드 블록으로 변환되는 패턴과 회피 방법:**

| 문제 패턴 | 원인 | 회피 방법 | 예시 |
|-----------|------|-----------|------|
| `` `텍스트` `` (백틱) | Markdown 인라인 코드 문법 | 백틱 제거, 괄호로 대체 | `` `AL2023` `` → `AL2023` 또는 `(AL2023)` |
| `18:00` (시간 형식) | 콜론 포함 문자열이 코드로 해석 | 한글로 표기 | `18:00` → `18시 00분` |
| 단독 숫자 (`1`, `2`, `3`) | 짧은 숫자가 코드로 해석 | 텍스트와 결합 | `1` → `Step 1` 또는 `1단계` |
| 단독 `-` (하이픈) | 빈 셀 대용으로 사용 시 strip되거나 오작동 | 한글 텍스트로 대체 | `-` → `해당없음` 또는 `이후 연속` |
| 빈 셀 | 빈 테이블 셀 | 공백 한 칸 또는 텍스트 | (빈칸) → ` ` 또는 `없음` |

**테이블 셀 작성 원칙:**
1. 테이블 셀에 단독 숫자, 단독 기호, 콜론 포함 문자열을 넣지 않는다
2. 셀 내용은 가급적 2글자 이상의 한글 텍스트로 작성한다
3. 시간은 `HH시 MM분` 형식을 사용한다
4. 빈 셀에는 `해당없음`, `없음`, `유지` 등 명시적 텍스트를 넣는다
5. 백틱(`` ` ``)은 테이블 안에서 사용하지 않는다

## 게시 전 검증 항목

1. Summary가 비어있지 않은지
2. Description이 Jira 최대 길이(32,767자) 초과하지 않는지
   - 초과 시: 요약본만 Description에, 전체는 첨부파일로
3. 프로젝트 키가 사용자에게서 명시적으로 받은 값인지
4. Label이 프로젝트에 존재하는지 (없으면 자동 생성)
5. 기존 이슈 업데이트인지 신규 생성인지 사용자에게 확인
6. AI 작성 안내 텍스트가 Description 최상단에 포함되었는지
8. `AI-Claude` 레이블이 labels 목록에 포함되었는지
7. **테이블 셀 위생 검증** (ADF 변환 함정 방지):
   - 백틱(`` ` ``)이 테이블 안에 없는지
   - 콜론 포함 시간 형식(`HH:MM`)이 테이블 셀에 없는지 → `HH시 MM분`으로 변환
   - 단독 숫자(`1`, `2`, `3`)가 테이블 셀에 없는지 → 텍스트 결합
   - 단독 하이픈(`-`)이 테이블 셀에 없는지 → 한글 텍스트로 대체
   - 빈 테이블 셀이 없는지 → 명시적 텍스트로 채움

## 업로드 후 보고 형식

작업 완료 후 다음을 보고한다:
- 생성/수정된 이슈 키 (예: MZ00006-1285)
- 이슈 URL
- 어떤 문서가 어디에 매핑되었는지 요약
