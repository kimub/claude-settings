## 작업계획서 품질 기준

### 공식 문서 구조 (에이전트 필수 참조)

모든 작업계획서는 아래 구조를 따른다. workplan-planner 에이전트는 이 구조를 그대로 사용해야 한다.

```markdown
# {작업명} 작업계획서

## 1. 개요

| 항목 | 내용 |
|------|------|
| 작업명 | {작업명} |
| 작업 일시 | {YYYY-MM-DD HH:mm ~ HH:mm} |
| 작업 목적 | {목적} |
| 작업자 | {작업자} |
| 참고 문서 | {관련 plan.md, design.md 등} |

## 2. 사전 준비

- [ ] {사전 준비 항목 1}
- [ ] {사전 준비 항목 2}

## 3. 작업 순서

### 3.1 {단계명} (예상 소요: {N}분)

**작업 내용:**
{구체적 명령어 또는 절차}

**검증:**
{이 단계의 검증 방법}

### 3.2 {다음 단계} ...

## 4. 검증

### 4.1 기능 검증
- {검증 항목}

### 4.2 성능/안정성 검증
- {검증 항목}

## 5. 롤백 절차

### 5.1 롤백 판단 기준
- {어떤 상황에서 롤백하는지}

### 5.2 롤백 순서
- {역순 롤백 절차}

## 6. 완료 보고

| 항목 | 내용 |
|------|------|
| 작업 결과 | 성공 / 실패 |
| 실제 소요 시간 | {N}분 |
| 특이사항 | {있으면 기재} |
```

### 작성 원칙

- 명령어는 즉시 실행 가능한 상태로 작성 (플레이스홀더 금지)
- 매 단계마다 검증 방법 필수 포함
- 롤백 절차 필수 (금융권 변경관리 요건)
- 민감 정보는 {SECRET_NAME} 변수 처리
- 시간 추정을 각 단계에 명시

### 검증 방법 유형

단계별 검증에는 아래 유형 중 적합한 것을 사용한다:

- **상태 확인**: 명령어 실행 후 상태 조회 (예: `kubectl get pods`, `systemctl status`)
- **로그 확인**: 에러/경고 로그 부재 확인 (예: `tail -f /var/log/app.log`)
- **응답 검증**: API 호출로 정상 응답 확인 (예: `curl -s http://localhost/health`)
- **데이터 검증**: 쿼리로 데이터 정합성 확인 (예: `SELECT COUNT(*) FROM ...`)
- **비교 검증**: 변경 전후 값 비교 (예: diff, 버전 비교)
- **접속 테스트**: 서비스 접근 가능 여부 확인 (예: 브라우저 접속, ping)

### 롤백 패턴 예시

유형별 롤백 패턴:

- **DB 변경**: 백업 복원 또는 역방향 마이그레이션 SQL
- **설정 변경**: 원본 설정 파일 복원 (사전 백업 필수)
- **배포**: 이전 버전 이미지/빌드로 재배포
- **인프라 변경**: Terraform state 복원 또는 이전 리소스 재생성
- **DNS/네트워크**: 이전 레코드/라우팅 규칙 복원

### plan.md에서 추출할 항목
- 작업 목적, 배경, 범위, 성공 기준

### design.md에서 추출할 항목
- 대상 시스템, 기술 구현 방안, 명령어, 설정값, 아키텍처

### 문체
- 간결하고 전문적, 정중한 톤 ("~합니다" 체)
- 기술 용어 일관성 유지
